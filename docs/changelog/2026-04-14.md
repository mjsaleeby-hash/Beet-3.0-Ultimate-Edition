# Changelog — 2026-04-14

## New Features

### Restore from Previous Versions

Files overwritten by a backup can now be recovered through a built-in versioned archive system.

- **Automatic archiving on overwrite** — before any overwrite (Skip Existing, Mirror-changed, and Replace paths in `TransferService`), `VersioningService.ArchiveBeforeOverwrite` saves the existing destination file to a versioned archive folder. If archiving fails, the copy is **skipped** rather than overwriting an unprotected file.
- **Previous Versions dialog** — right-click any file or folder in any of the three file list panes and choose **Previous Versions…** to open a 640×500 modal showing a timestamped list of archived copies. Columns: Archived, Size, Archive Location.
- **Double-click to restore** — double-clicking an entry in the dialog restores that version in place.
- **Delete archived version** — removes the selected archived copy from the store without affecting the live file.
- **`VersioningService`** — core service; methods: `ArchiveBeforeOverwrite` (bool return), `FindVersionsRoot`, `GetArchivedVersions`, `RestoreVersion`, `DeleteArchivedVersion`.
- **`ArchivedVersion` model** — sealed record with `ArchivedPath`, `OriginalPath`, `ArchivedAt`, `Size`, `DisplayTimestamp`, `DisplaySize`.
- Versioning is enabled and configured (max versions) per job via the wizard Advanced step and the Schedule dialog.

**New files:** `Models/ArchivedVersion.cs`, `Services/VersioningService.cs`, `ViewModels/PreviousVersionsViewModel.cs`, `Views/PreviousVersionsDialog.xaml`, `Views/PreviousVersionsDialog.xaml.cs`

### Extract Archive (.zip)

ZIP archives can now be extracted directly from within the app — no external tool needed.

- **Extract Here** — right-click a `.zip` file and choose **Extract Here** to extract its contents into a sibling folder with the same base name.
- **Extract To…** — right-click and choose **Extract To…** to pick any destination folder via a folder browser dialog.
- **ZIP-slip protection** — every entry path is canonicalized and checked to confirm it remains inside the target directory; entries that would escape are silently skipped.
- **Full transfer integration** — extraction honors cancel/pause signals and reports progress identically to a standard file copy.
- **Sharing-violation handling** — files locked by another process are caught and recorded as file errors rather than aborting the entire extraction.
- Implemented as `TransferService.ExtractAsync`; `MainViewModel` exposes `ExtractHereAsync`, `ExtractToAsync`, and `TryResolveArchive`.

### Pre-flight Disk Space Preview

Before committing to a backup, the app now estimates whether the destination drive has sufficient free space.

- **`DiskSpaceService.Preview`** — static, best-effort; never throws. Sums source sizes (respecting exclusions), applies a **0.7× compression estimate** for zip jobs, and compares against available free space.
- **Status levels:** `Sufficient` / `Tight` (remaining free space after transfer < 10% of total drive) / `Insufficient` / `Unknown` (UNC/network paths).
- **Wizard summary banner** — a colored banner appears above the summary card: red for Insufficient, amber for Tight, neutral for informational.
- **Wizard gate** — clicking Finish when status is Insufficient shows a Yes/No confirmation before proceeding.
- **Schedule dialog banner** — an inline warning appears below the Estimate Size row.
- **Schedule dialog gate** — clicking Schedule when status is Insufficient requires confirmation.
- `WizardStepSummaryViewModel` exposes `DiskSpaceMessage`, `HasDiskSpaceMessage`, `IsInsufficientSpace`, `IsTightSpace`, `LastPreview`.
- `ScheduleDialogViewModel` exposes the same properties.

**New file:** `Services/DiskSpaceService.cs`

### Versioning and Compression Options in Wizard and Schedule Dialog

The wizard's Advanced step and the Schedule dialog now expose versioning and compression settings.

- **Enable Versioning** — when enabled, a **Max Versions** field appears (default: 5). Versioning and Compression are mutually exclusive — enabling one disables the other.
- **Enable Compression** — compress the backup output to a `.zip` archive.
- Both settings flow through `BuildJob()` in `BackupWizardViewModel` and are reflected in the wizard summary ("Keep N versions" / "Compress to .zip").
- `ScheduleDialogViewModel` applies the same mutual exclusion and normalization.

### Scheduler Survives Reboot — Windows Task Scheduler Integration

Scheduled backups now persist across reboots through Windows Task Scheduler, eliminating the requirement that the app must be running for a job to execute.

- **`WindowsTaskSchedulerService`** — wraps `schtasks.exe` (no NuGet dependency, no COM interop).
  - Task naming: `BeetsBackup_{GuidN}` (compact GUID, no dashes).
  - `Register(job)` — creates or replaces the task with `/F`; runs as the interactive user at `RL LIMITED`.
  - `Unregister(jobId)` — deletes the task.
  - `ReconcileAll(jobs)` — re-registers all enabled jobs; called at startup.
  - Schedule mapping: Daily → `/SC DAILY`, Weekly → `/SC WEEKLY`, Every N hours → `/SC HOURLY /MO N`, sub-hour → `/SC MINUTE /MO N`, one-time → `/SC ONCE`.
  - Past-due `NextRun` times are bumped to now + 1 minute to avoid immediate double-run.
- **`SchedulerService` integration:**
  - `AddJob` calls `WindowsTaskSchedulerService.Register` (best-effort — failure is logged, job still saved).
  - `RemoveJob` calls `WindowsTaskSchedulerService.Unregister`.
  - New `RunJobByIdAsync(Guid)` — executes a job headlessly; advances `NextRun`; used by the CLI entry point below.
  - New `ReconcileWindowsTasks()` — re-registers all enabled jobs; called on a background thread at app startup.
- **Headless CLI mode (`--run-job`)** — Windows Task Scheduler launches the app with `--run-job {guid}` or `--run-job={guid}`. The app detects this flag before the single-instance mutex, runs the specified job with minimal DI and no window, then exits with code `0` (success), `1` (job not found), or `2` (job failed).
- **`CliArgs.TryParseRunJob`** — parses both space-separated and `=`-delimited forms; case-insensitive.
- **In-process `PeriodicTimer`** is retained as a safety net for jobs running while the app is open interactively.

**New files:** `Services/WindowsTaskSchedulerService.cs`, `Helpers/CliArgs.cs`
